## 三、技术架构

```
用户浏览器 (HTTPS :443)
  ↓ Cloudflare CDN
[Nginx] ← TLS 终止 + 反向代理
  ↓
[PM2] ← 进程守护 (fork mode)
  ↓
Next.js server.js (standalone build, :3000)
  ↓              ↓
SQLite (WAL)   Cloudflare R2 (图片)
```

| # | 决策 | 替代方案 | 理由 |
|---|------|---------|------|

## 四、重点难点 & 解决方案

### 难点 1：CoT 模型的 output 数组陷阱 ⚠️

**现象**：豆包 AI 返回成功但 `output[0].content[0].text` 是 `undefined`。

**根因**：CoT（Chain of Thought）模型的 `output` 数组里，`output[0]` 是推理过程（reasoning），`output[1]` 才是实际消息。

**解决**：
```typescript
// ❌ 错误：直接取 output[0]
const text = data.output[0].content[0].text; // undefined

// ✅ 正确：找到 type === "message" 的 item
const messageItem = data.output.find(o => o.type === "message");
const text = messageItem.content.find(c => c.type === "output_text").text;
```

### 难点 2：Sharp 跨平台二进制不兼容 🔧

**现象**：macOS 上 build 的 standalone 部署到 Linux VPS 后，`Could not load the "sharp" module using the linux-x64 runtime`。

**根因**：本地构建时 Next.js 把 macOS 版 sharp 二进制打包进了 `.next/node_modules/sharp-*` 缓存目录。rsync 到 Linux 后无法加载。

**解决**：
1. VPS 上 `npm install --force --platform=linux --arch=x64 sharp`
2. 把 Linux sharp 的 `build/` `vendor/` `install/` 目录复制到 `.next/node_modules/sharp-<hash>/` 替换 macOS 版本
3. 自动化：写成 `vps-setup.sh`，每次部署自动执行

### 难点 3：Prisma SQLite 路径解析 🗺️

**现象**：数据库有 12 张照片但 API 返回 1 张——连到了错误的数据库文件。

**根因**：`DATABASE_URL=file:./dev.db` 中的 `./` 相对于 `prisma/schema.prisma` 所在目录解析。实际路径是 `/home/bldcam/prisma/dev.db`（测试库），而真实数据在 `/home/bldcam/dev.db`。

**解决**：改用绝对路径 `DATABASE_URL=file:/home/bldcam/dev.db`。本地 `.env` 用相对路径，deploy.sh 自动修正为绝对路径。

这个坑踩了两次——第一次修好后又因为部署覆盖 `.env.production` 而复现。

### 难点 4：SSH 断连导致部署半途而废 💔

**现象**：`npm install` 跑太久，SSH 连接断开（exit 255），rsync 已清掉旧 node_modules 但新文件未完整写入 → `Cannot find module 'next'` → 502。

**根因**：原来的 `deploy.sh` 是一整条长 SSH 命令：`ssh ... 'npm install && pm2 restart...'`。任何网络波动都导致全流程失败。

**解决**：
1. 抽离 `vps-setup.sh` 到 VPS 独立脚本
2. `deploy.sh` 改为：rsync → scp 脚本 → `nohup bash vps-setup.sh &`（1 秒 SSH，fire-and-forget）
3. deploy.sh 轮询 PM2 状态等待上线（最多 120s）

```
Before: rsync → ssh "长命令" → 💥断连→失败
After:  rsync → scp 脚本 → nohup & → 轮询 PM2 → ✅
```

### 难点 5：npm install 中断的连锁损坏 🔗

**现象**：SSH 断开时 npm 正在重命名包目录，`.临时目录` 残留。后续 `npm install --force` 报 `ENOTEMPTY: directory not empty, rename`。去掉 `--force` 后 npm 连锁更新依赖，把 `next` 也删了。

**解决**：
1. `find node_modules -maxdepth 1 -name ".*" -type d -exec rm -rf {} +` 清临时目录
2. 如果已损坏：`rm -rf node_modules && npm install --production`
3. 显式安装 `@img/sharp-libvips-linux-x64`（可选依赖不自动装）

### 难点 6：Feed 图片 CLS 跳动 📐

**现象**：Feed 页面加载时图片尺寸从 1920×1080（fallback）跳到真实尺寸，用户看到明显抖动。

**根因**：`useImageDisplaySize` 用 `useState({width:0, height:0})`，首次渲染无真实 viewport，用 fallback 1920×1080 计算尺寸。

**解决**：`useState(getViewport)` 同步初始化——首次渲染前直接读 `window.innerWidth/Height`，零帧延迟，CLS 从 0.15 降到 0。

### 难点 7：N 个组件各创建 resize 监听器 🔁

**现象**：Feed 每张卡片都 `window.addEventListener("resize", update)`，20 张照片 = 20 个监听器。

**解决**：模块级 singleton 订阅器：
```typescript
const _resizeCbs = new Set<ResizeCb>();
function subscribeResize(cb) {
  _resizeCbs.add(cb);
  if (_resizeCbs.size === 1) window.addEventListener("resize", _onResize);
  return () => { /* cleanup */ };
}
```
120ms debounce，所有组件共享 1 个监听器。

### 难点 8：iOS 安全区域适配 📱

**现象**：全站零 `env(safe-area-inset-*)`，iPhone 的刘海/Dynamic Island/Home Indicator 会遮挡内容。

**解决**：6 个文件加 `env(safe-area-inset-*, 0px)` — 关键是用 `0px` fallback，桌面端求值为 0，完全不影响 PC。

| 文件 | 修复 |
|------|------|
| `globals.css` | body `padding-bottom` |
| `header.tsx` | `pt-[env(safe-area-inset-top,0px)]` |
| `admin-console.tsx` | 后台 header 同上 |
| `city-sidebar.tsx` | nav `paddingTop`/`paddingBottom` env |
| `photo-map.tsx` | 地图高度 `calc(100svh - 4rem - env(...) - env(...))` |
| `photo-lightbox.tsx` | EXIF overlay `pb-[calc(1rem+env(...))]` |

### 难点 9：手机端触屏按钮不可见 👆

**现象**：后台编辑/删除按钮用 `opacity-0 group-hover:opacity-100`，手机没有 hover → 按钮永远不可见。

**解决**：
```html
<!-- 手机始终可见，桌面端保持 hover 显示 -->
<div class="opacity-100 md:opacity-0 md:group-hover:opacity-100">
```

### 难点 10：rsync --delete 清掉数据库 🗑️

**现象**：部署后 `dev.db` 消失。

**解决**：deploy.sh 加排除规则：
```bash
rsync --delete \
  --exclude='dev.db' \
  --exclude='ecosystem.config.cjs' \
  --exclude='logs/' \
  --exclude='backups/' \
  ...
```

### 难点 11：PM2 环境变量缓存 ♻️

**现象**：改了 `.env.production` 的 `DATABASE_URL`，`pm2 restart --update-env` 不生效。

**解决**：改用 `pm2 delete bldcam && pm2 start server.js --name bldcam`——先删后启，彻底刷新环境变量。

### 难点 12：腾讯云 22 端口拦截 🔒

**现象**：SSH TCP 握手成功但收到空横幅，`kex_exchange_identification: Connection closed`。

**解决**：sshd 加监听 2222 端口，所有 SSH 连接切到 2222。

### 难点 13：PM2 systemd 未激活 🔌

**现象**：`pm2-ubuntu.service` 状态 `inactive (dead)`，VPS 重启后网站不会自动恢复。

**解决**：
```bash
pm2 save
sudo systemctl enable --now pm2-ubuntu
```

## 五、经验教训

### 做得好的 ✅

1. **先闭环再优化**：Day 1 就把上传→存储→展示跑通，之后所有改进都是增量，每个功能独立可回退
2. **失败快速切换**：MiMo API 不行就清理换豆包，不纠结
3. **SQLite 够用不升级**：省掉的运维时间是实打实的
4. **随时存档**：记忆文件让跨会话上下文不丢失，是 solo 开发者的救命稻草
5. **部署独立脚本**：vps-setup.sh 把部署从"祈祷 SSH 别断"变成"fire and forget"
6. **零侵入式修复**：`env(safe-area-inset-*, 0px)` 的 `0px` fallback 保证桌面端完全不受影响
7. **用 AI 但不依赖 AI**：豆包做自动打标，但上传流水线 fire-and-forget 调用，AI 挂掉不影响上传

### 踩过的坑 🕳️

1. **macOS build → Linux deploy = 定时炸弹**：Sharp、Prisma、所有 native 模块都需要双平台处理
2. **环境变量是无声杀手**：PM2 缓存、相对路径解析、Next.js 构建时 vs 运行时——一个小错排查半天
3. **rsync --delete 是双刃剑**：清理方便但一不留神删数据库
4. **npm install 不能打断**：中断后的 `.临时目录` 残留会导致连锁损坏
5. **CoT 模型的 output 不是 Chat 模型**：`output.find(o => o.type === "message")` 而不是取 `output[0]`
6. **相对路径在 Prisma 中不直观**：`file:./dev.db` 相对 schema 文件位置，不是 cwd

### 可以更好的 🔮

1. **测试**：目前零自动化测试，全靠眼睛看
2. **错误边界**：`error.tsx` 还没有，组件崩了整个页面白屏
3. **上传断点续传**：大文件断网得重来
4. **代码组织**：`formatExifLine` 定义了两次，应抽到 `lib/format.ts`
5. **地图 marker 聚合**：Supercluster 已安装未接入，上百个点会卡

## 八、写给自己的话

这是你的第一个全栈项目——从 `create-next-app` 一行命令开始，到最终挂上域名被全世界访问。

你学到了：
- **Linux 服务器管理**：从零到能装环境、配 Nginx、看日志、排查进程
- **DNS + CDN**：Cloudflare 域名托管、A 记录、代理模式、SSL
- **SSH + 远程部署**：Key 认证、rsync 增量同步、端口转发
- **跨平台编译**：Prisma binaryTargets、Sharp 平台差异、native module
- **故障排查**：看日志→定位→假设→验证→修复 的完整链路
- **AI 编程**：不是让 AI 替你写代码，而是让 AI 做你的 senior engineer reviewer

最重要的是——你做到了"从零搭建不是复制"。参考 camlife 的审美但不 fork，每个组件自己写，理解每一行。

这个项目的哲学：**简洁优先，不做 SaaS，不引入不必要复杂度**。希望你以后的项目也能保持这个心态。
